## 5. Перенос авторизации на реальный бэкенд (главный этап)

### 5.1 Перенаправление авторизации на `localhost:5173`
**Запрос:** «Ты должен был работать с http://localhost:5173. Сделай, чтобы при авторизации через почту всё авторизовывалось в http://localhost:5173 и пользователи создавались там»

**Что было сделано:**
- Создана полноценная система авторизации на PHP-бэкенде в директории `/api/`:

#### Созданные API-файлы:
| Файл | Назначение |
|------|-----------|
| `api/config.php` | Константы подключения к БД, OAuth-ключи, `FRONTEND_URL = 'http://localhost:5173'` |
| `api/db.php` | PDO-подключение к MySQL, JWT/Bearer-аутентификация, вспомогательные функции (`currentUser()`, `getBearerToken()`, `handleSocialLogin()`, `jsonResponse()`) |
| `api/auth/login.php` | Авторизация по email + пароль (POST), выдача API-токена |
| `api/auth/register.php` | Регистрация нового пользователя (POST), хеширование пароля, выдача API-токена |
| `api/auth/logout.php` | Выход из системы (очистка API-токена) |
| `api/user/me.php` | Получение данных текущего пользователя по Bearer-токену |

#### Изменения во фронтенде:
| Файл | Изменения |
|------|-----------|
| `src/app/state/PortalContext.tsx` | Переписаны функции `login()`, `logout()`, `registerUser()` — теперь все работают через `fetch()` к реальному PHP API вместо мок-данных |
| `src/app/pages/LoginPage.tsx` | Форма авторизации отправляет данные на `api/auth/login.php`, токен сохраняется в `localStorage` |
| `src/app/pages/RegisterPage.tsx` | Форма регистрации отправляет данные на `api/auth/register.php` |

#### Структура базы данных (MySQL):
- Таблица `users`: `id`, `email`, `password_hash`, `name`, `avatar`, `email_verified`, `role_id`, `api_token`, `specialization`, `is_young_specialist`, `is_first_employment`
- Таблица `roles`: `id`, `name` (Пользователь, Модератор контента, Администратор организации, Суперадминистратор)
- Таблица `social_accounts`: `id`, `user_id`, `provider`, `provider_user_id`, `provider_email` — привязка соцсетей

### 5.2 Перенос OAuth из тестового проекта в основной
**Запрос:** «Без эмуляций нужно перенести авторизацию через площадки из тестового проекта в наш реальный, а тестовый удалить»

**Что было сделано:**
- Созданы OAuth-файлы в директории `/api/auth/`:
  - `api/auth/google.php` — редирект на Google OAuth, а после callback — создание/привязка пользователя в MySQL и перенаправление на фронтенд с токеном.
  - `api/auth/yandex.php` — аналогичная логика для Яндекса.
- Callback-URL-ы настроены с учётом кириллического пути (`дипломчики` → `%D0%B4%D0%B8%D0%BF%D0%BB%D0%BE%D0%BC%D1%87%D0%B8%D0%BA%D0%B8`).
- Во фронтенде `LoginPage.tsx` кнопки OAuth ведут на `http://localhost/дипломчики/api/auth/google.php` и `yandex.php`.
- Фронтенд (`LoginPage.tsx`) обрабатывает возврат с OAuth: считывает токен из URL-параметра и сохраняет в `localStorage`.

### 5.3 Отладка Google OAuth
**Запрос:** «Missing required parameter: client_id — Error 400: invalid_request при попытке войти через Google»

**Что было сделано:**
- Обнаружена проблема: `config.php` содержал пустые заглушки вместо реальных Client ID / Secret.
- Вставлены реальные ключи Google OAuth в `api/config.php`.

**Запрос:** «This app doesn't comply with Google's OAuth 2.0 policy — Error 400»

**Что было сделано:**
- Объяснена причина: кириллический `redirect_uri` с путём `/дипломчики/` вызывал отклонение Google.
- Выдана инструкция: в Google Cloud Console необходимо добавить URI с URL-encoded кириллицей:
  `http://localhost/%D0%B4%D0%B8%D0%BF%D0%BB%D0%BE%D0%BC%D1%87%D0%B8%D0%BA%D0%B8/api/auth/google_callback.php`

**Запрос:** «Error 400: redirect_uri=http://localhost/дипломчики/api/auth/google_callback.php»

**Что было сделано:**
- Подтверждена необходимость URL-encode кириллического пути в Google Cloud Console.
- Выдана пошаговая инструкция по настройке Authorized Redirect URIs.

**Запрос:** «Проверь в базе данных какие пользователи зарегистрированы, их email-ы и пароли, проверь дубликаты, объясни ошибку Invalid email or password»
